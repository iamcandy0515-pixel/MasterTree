# 🧩 MasterTreeApp 개발 표준 및 작업 규칙 (DEVELOPMENT_RULES.md)

이 문서는 MasterTreeApp 프로젝트의 코드 품질 유지, 소스 유실 방지, 그리고 안정적인 개발 환경(Windows 11)을 보장하기 위한 **필수 준수 규칙**입니다.

## 0. 작업 전제 조건 및 환경 설정

- **0-1. [Git 백업]** 모든 작업 시작 전, 현재 상태를 GitHub 또는 로컬 Git에 반드시 백업(stash/commit)한다.
- **0-2. [Windows 11 환경 최적화]**
    - 터미널 명령어 실행 전 반드시 인코딩 설정(`chcp 65001`)을 확인하여 한글 깨짐을 방지한다.
    - 파일 경로 길어짐(Long Path) 이슈를 고려하여 절대 경로를 정확히 활용한다.
- **0-3. [오류 사전 예방]** Import 에러, 린트(Lint) 에러, UI Overflow 에러 등을 설계 단계에서 미리 예측하고 예방하는 로직을 구현한다.
- **0-4. [소스 정합성 및 유실 방지]**
    - 작업 완료 후 GitHub의 백업본과 현재 수정한 소스의 차이(diff)를 분석한다.
    - 수정 대상 외의 코드가 의도치 않게 삭제되거나 불필요한 코드가 추가되지 않았는지 정합성을 최종 체크한다.

## 1. 코드 구조 및 모듈화 (Source Splitting)

- **1-1. [200줄 제한 원칙]** 단일 파일 소스가 **200줄**을 초과할 경우, 반드시 기능별/위젯별 소스 분리 작업을 진행한다.
- **1-2. [효율적 통신 및 콜백]** 소스 분리 시 부모-자식 간의 통신은 콜백(Callback)을 적절히 사용하여 로드 부하나 로직 오류를 유발하지 않아야 한다.
- **1-3. [분리 후 에러 체크]** 소스 분리 시 발생할 수 있는 Import 경로 문제(부모/자식), 린트 에러, UI 구조 붕괴(Overflow)를 사전에 체크하여 예방한다.

## 2. 작업 프로세스 (Task Workflow)

- **2-1. [작업 계획서 및 To-Do 작성]**
    - 작업 시작 전 `docs/plan/{task-slug}.md` 파일에 **작업 계획서**와 **To-Do List**를 작성한다.
    - 진행 상황에 따라 To-Do List를 하나씩 체크하며 작업 누락(Skip)을 원천 차단한다.
- **2-2. [Socratic Gate 통과]** 복잡한 요청의 경우, 구현 전 최소 3가지 이상의 전략적 질문을 통해 요구사항을 명확히 정의한다.
- **2-3. [컴파일 및 빌드 완결성]** 소스 수정 완료 후 단순히 코드만 작성하는 것이 아니라, 실제 컴파일 및 빌드 시 발생하는 모든 이슈를 제거하는 작업까지 마무리한다.

## 3. 코드 품질 (Clean Code)

- **3-1. [Clean Code 준수]** `@[skills/clean-code]` 규칙을 따르며, 과도한 엔지니어링을 피하고 직관적인 코드를 작성한다.
- **3-2. [린트 및 보안 체크]** 모든 작업 종료 전 `flutter analyze`와 보안 스캔 스크립트를 실행하여 'Critical' 블로커를 해결한다.

## 4. 빌드 및 인프라 설정 (Build & Infrastructure - Flutter 3.7.12)

- **4-1. [안드로이드 빌드 환경 고정 (Flutter 3.7.12 대응)]** 빌드 안정성과 구버전 플러그인 호환성을 위해 아래 버전을 고수하며 **절대 임의 변경하지 않는다.**
    - **Flutter**: `3.7.12` / **Dart**: `2.19.6`
    - **Java**: `11` (LTS)
    - **Gradle**: `7.5` (gradle-wrapper.properties)
    - **Android Gradle Plugin (AGP)**: `7.2.0` (build.gradle)
    - **Kotlin**: `1.7.10` (build.gradle)
- **4-2. [버전 상향 금지 및 에러 메시지]** 위 버전을 초과하여 업데이트를 시도할 경우, 빌드 시나 에디터 상에서 명확한 에러 메시지를 발생시켜 작업을 중단시켜야 한다.
- **4-3. [의존성 충돌 해결 (Resolution Strategy)]** 특정 라이브러리가 상위 AGP 버전을 요구하여 빌드가 실패할 경우, 무리한 AGP 업그레이드 대신 `app/build.gradle`의 `configurations.all { ... }` 블록을 통해 안정적인 하위 호환 버전을 강제(force)하여 빌드 정합성을 유지한다. (예: `androidx.core:core:1.10.1`, `androidx.activity:activity:1.7.2` 등)
- **4-4. [웹 관련 코드 격리 (Web Independence)]** `dart:html` 또는 `dart:ui_web`을 직접 사용하지 않는다. 반드시 `WebUtils` 추상화 레이어를 사용하여 모바일 빌드 시 발생하는 컴파일 에러를 원천 차단한다.

**주의:** 본 규칙을 어기는 작업 방식은 용납되지 않으며, 모든 에이전트(Antigravity)는 이 규칙을 최우선 순위로 설정한다.
