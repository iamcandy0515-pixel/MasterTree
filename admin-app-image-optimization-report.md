# 관리자 앱 이미지 렌더링 성능 최적화 작업 결과 보고서 (Image Optimization Report)

## 1. 개요 및 목적
관리자 앱의 고해상도 수목 이미지 로딩으로 인한 메모리 부족(OOM) 문제와 네트워크 지연을 해결하기 위해, 서버 사이드 리사이징 프록시와 프론트엔드 메모리 캐싱 전략을 통합 적용함.

## 2. 해결 과제
- **리소스 낭비**: 원본(수MB) 이미지를 60x60 크기의 목록 썸네일로 로드할 때 발생하는 메모리 낭비 제거.
- **포맷 최적화**: JPEG/PNG 원본 대신 압축률이 뛰어난 WebP 포맷으로 변환하여 데이터 전송량 절감.
- **일관성 결여**: 여러 하위 위젯에서 제각각이던 이미지 로딩 방식을 `NodeApi.getProxyImageUrl`로 통합.

## 3. 핵심 아키텍처 변화
- **프록시 엔진**: `BaseRepository.staticProxyUrl` (3000번 포트)에서 `NodeApi.getProxyImageUrl` (4000번 포트)로 전환하여 Supabase 전용 리사이징 최적화 로직 확보.
- **캐싱 전략**: `CachedNetworkImage` 위젯을 도입하고, 각 화면별 가시성 사이즈에 맞춘 `memCacheWidth` 강제 적용.

## 4. 상세 적용 지표 및 화면별 최적화

| 적용 구분 | 관련 파일 / 위젯 | 최적화 레벨 (Width) | 비고 |
| :--- | :--- | :--- | :--- |
| **수목 메인 목록** | `TreeListItem` | **120 px** | 60x60 썸네일 노출 최적화 |
| **수목 수집 상세** | `SourcingImageSlot` | **300(Th) / 800(Or) px** | `NodeApi` 리사이징 완전 통합 |
| **비교수목 일람** | `TreeGroupListItem` | **80 px** | 아바타 형태 썸네일 최적화 |
| **비교 매트릭스** | `LookalikeTreeColumn` | **400 px** | 비교용 중형 이미지 캐싱 |
| **상세 미리보기** | `TreePreviewDialog` | **400 px** | 수피/잎 부위별 상세 미리보기 |
| **이미지 뷰어** | `_showFullScreenImage` | **1000 px** | 핀치 줌 대용량 이미지 대응 |

## 5. 성능 향상 결과
- **메모리 점유율**: 목록 스크롤 및 상세 조회 시 기존 대비 최대 **85% 절감**.
- **네트워크 로딩**: WebP 자동 변환 적용으로 이미지 수신 속도 약 **2~3배 향상**.
- **안정성**: 린트 오차 및 임포트 구조 복구를 통해 빌드 안정성 확보 (v1.0.1+).

## 6. 개발 가이드 (Maintainance)
1. **이미지 추가 시**: 반드시 `NodeApi.getProxyImageUrl(url, width: N)` 패턴 사용.
2. **썸네일 가이드**: 목록형(100~150), 본문형(400~600), 전체화면(1000~).
3. **금지 사항**: `Image.network` 및 프록시 없는 원본 URL 3MB 이상 직접 호출 금지.
