# 구글 드라이브 인증 방식 통합 작업 계획서 (JWT 일원화)

개발자님, 현재 프로젝트에 혼재된 3가지 인증 방식을 **서비스 계정(JWT) 인증 방식**으로 일원화하여 유지보수 효율성과 보안성을 높이기 위한 작업 계획서입니다.

## 1. 현황 및 문제점 분석

현재 시스템은 다음과 같은 방식으로 드라이브에 접근하고 있습니다:
- **방식 A (API Key)**: `GoogleDriveFileService`에서 공개 파일 다운로드 시 사용. 보안이 취약하고 권한이 제한적임.
- **방식 B (OAuth2)**: 기존 관리자 로그인 기반 연동. Refresh Token 만료 시 주기적인 재인증이 필요함.
- **방식 C (JWT / 서비스 계정)**: 이번에 도입된 방식. 서버-대-서버 통신에 최적화되어 있으나 일부 스크립트에서만 개별적으로 사용 중.

## 2. 통합 목표
> **"모든 구글 드라이브 접근을 서비스 계정(JWT) 인증으로 일원화한다."**

## 3. 상세 작업 단계

### [1단계] 공통 인증 서비스 강화 (`GoogleDriveAuthService.ts`)
- [x] 서비스 계정(JWT) 지원 로직 추가 (완료)
- [ ] OAuth2 관련 코드 제거 또는 Legacy용으로 분리 및 `@deprecated` 마킹
- [ ] 인증 실패 시 공통 에러 핸들링 로직 추가

### [2단계] 개별 서비스 및 컨트롤러 수정
- **`GoogleDriveFileService.ts`**:
    - [ ] 생성자에서 API Key Fallback 로직 제거
    - [ ] `downloadFile` 메서드에서 항상 인증된(Authenticated) 스트림을 사용하도록 강제
- **`GoogleDriveService.ts`**:
    - [ ] 이미지 검색 및 링크 생성 로직에서 서비스 계정 권한 확인 로직 강화

### [3단계] 유틸리티 스크립트(`src/scripts/`) 전면 개편

- [ ] 모든 스크립트가 `googleDriveAuthService`를 참조하도록 수정

## 4. 기대 효과
- **안정성**: 토큰 만료로 인한 서비스 중단 원천 차단
- **유지보수**: 인증 관련 코드가 하나로 통합되어 버그 수정 및 관리 용이
- **보안**: API Key 노출 위험 제거 및 최소 권한 원칙(POLP) 적용 가능
