# 사용자 맞춤형 로그인 및 인증 시스템 고도화 계획서 (v2)

## 1. 개요

신규 사용자는 가입 절차(이메일 포함)를 진행하고, 기존 사용자는 최소 정보(이름/휴대폰/입장코드)만으로 로그인하는 시스템을 구축합니다. 시스템 부하를 최소화하면서도 부드러운 UX를 제공하는 두 가지 방안을 제시합니다.

---

## 2. [방안 A] 실시간 동적 전환 방식 (기본 제안)

사용자가 입력하는 도중에 서버에서 신규/기존 여부를 판단하여 UI를 변경합니다.

### 워크플로우

- **동작**: 이름과 휴대폰 번호 입력을 완료(Focus 상실 또는 Debounce)하면 서버 API(`findUser`) 호출.
- **UI 변화**:
    - 기존 유저: 이메일 필드 숨김 유지.
    - 신규 유저: 이메일 필드가 애니메이션과 함께 노시.
- **장점**: 사용자가 자신이 신규인지 기존인지 고민할 필요 없이 시스템이 가이드함.
- **단점**: 입력 시마다 또는 특정 시점에 서버 요청이 발생하여 네트워크 트래픽 증가.

---

## 3. [방안 B] 로컬 캐싱 기반 '저부하' 최적화 방식 (추천 제안) 🚀

시스템 부하를 획기적으로 줄이면서도 정확도를 유지하는 합리적인 제안입니다.

### 핵심 전략: "Local Memory First, Lazy API Check"

1. **로컬 기억 장치 활용**:
    - 사용자가 과거에 한 번이라도 해당 기기에서 로그인에 성공했다면, `SharedPreferences`에 이메일 정보를 암호화하여 저장합니다.
    - 앱 기동 시 로컬에 정보가 있다면 즉시 **'기존 유저 모드'**로 시작하여 이메일 필드를 아예 그리지 않습니다. (서버 조회 0회)

2. **지연된 서버 확인 (Lazy Check)**:
    - 로컬 기록이 없는 경우에만 이름/번호 입력 후 **1회만** 서버에 확인을 요청합니다.
    - 서버 통신 횟수를 최소화하여 Supabase API 호출 비용 및 서버 부하를 절감합니다.

3. **입장코드 우선 검증**:
    - 신규/기존 여부와 상관없이 '입장코드'가 틀리면 로그인을 거부하므로, 인증의 첫 단계에서 보안을 유지하며 불필요한 DB 조회를 방지합니다.

---

## 4. 기술적 상세 구현 계획

### [Frontend] Flutter User App

- **`AuthController.dart`**:
    - `isFirstLaunch`: 로컬 저장소 확인용 변수 추가.
    - `debounce`: 이름/번호 입력 후 500ms 대기 후 서버 조회 기능 구현.
- **UI Component**:
    - 이메일 필드에 `Visibility` 또는 `AnimatedSwitcher` 적용.

### [Backend] Node.js API

- **`users` 테이블 최적화**:
    - `name`, `phone` 필드에 복합 인덱스(Composite Index) 생성하여 조회 속도 극대화.
    - 불필요한 전체 필드 조회 대신 `EXISTS` 쿼리 활용으로 부하 감소.

## 6. 기대 효과 및 리스크 관리

- **부하 감소**: 로컬 캐싱 도입으로 서버 호출 빈도를 약 60~80% 이상 절감 가능.
- **사용자 경험**: 재방문 시 즉각적인 로그인 폼 구성으로 빠른 진입 가능.
- **리스크**: 기기를 변경한 기존 사용자의 경우 '신규 유저'로 인식될 수 있으나, 이름/번호 입력 후 서버 확인 과정에서 즉시 '기존 유저'로 자동 전환되므로 문제없음.

---

## 7. 활동 유저 식별 고도화 (익명 로그인 대응) 🛠️

익명 로그인 방식에서도 '비활동 유저' 오분류 문제를 해결하기 위해 다음 로직을 추가 적용했습니다.

1. **Auth ID 동기화**:
    - 로그인 성공 시 생성된 익명 Auth ID를 `users` 테이블의 `auth_id` 컬럼에 실시간 업데이트.
    - 이를 통해 익명 세션과 실제 사용자 레코드('홍길동' 등)를 연결.

2. **데이터 기반 활동 집계**:
    - 관리자 대시보드의 '활동 유저' 판단 기준을 Auth 세션 시간이 아닌, `quiz_attempts` 테이블의 **최근 7일간 실제 학습 기록**으로 변경.
    - 사용자가 문제를 풀 때마다 `user_record_id`를 함께 저장하여 익명 로그인 상태에서도 정확한 사용자 식별 및 통계 실현.
